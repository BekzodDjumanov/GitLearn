Cherry Pick

Cherry Pick in Git is a feature that allows a developer to apply a commit to another designated branch, without merging the full branch history. It is a seamless way to infuse two branches without including past changes.

  • Works on a commit-by-commit basis.
  • Maintains the original commitâ€™s changes but creates a new commit ID in the target branch.
  • Useful when a bug fix or feature made in one branch is needed in another without merging all changes.

    • Conflicts can happen if changes overlap, just like in merges.

Common Commands

Apply a specific commit to the current branch

$ git cherry pick <commit-hash>

Apply a range of commits (excluding A, including B).

$ git cherry-pick A..B

Apply a range of commits including A and B

$ git cherry-pick A^..B

Apply changes without creating a commit yet

$ git cherry-pick --no-commit <commit-hash>

Best Practices

Be cautious

Cherry pick can lead to duplicate commits, which can lead to conflicts in merging. In some circumstances, git merge or git rebase is preferred.

Add proper comments

When committing (git commit -m "message"), utilize proper and sustainable comments to identify the details of each cherry pick.

Testing and validation

Always ensure that a program works as expected after a cherry pick, otherwise you may result in reverting or reseting the commit.

Summary

git cherry-pick copies a specific commit from one branch to another, without merging the whole branch.
