Forking Workflow

A forking workflow is a model commonly employed in open-source projects. Instead of pushing directly to the main repository, contributors fork the repository into their own account, make changes, and then submit a Pull Request to merge changes back into the original repository.

How it Works

  • 1. Fork the repository to your own GitHub account.
  • 2. Clone your fork locally.
  • 3. Create a branch for your feature or fix.
  • 4. Make changes and commit locally.
  • 5. Push the branch to your forked repository.
  • 6. Open a Pull Request to merge your branch into the original repository.

  • Open source projects with many contributors.

Common Workflow Command

# Fork the repo on GitHub, then clone it locally
$ git clone https://github.com/your-username/forked-repo.git

## Create a new branch
$ git checkout -b feature/add-login

# Commit changes
$ git add .
$ git commit -m "Add login feature"

# Push to your fork
$ git push origin feature/add-login

# Open Pull Request on GitHub

Advantages

  • Protects the main repository from unapproved changes.
  • Allows many contributors to work independently.
  • Enables structured code review via Pull Requests.

Disadvantages

  • Slightly more steps than direct branch workflows.
  • Contributors must manage their own fork and keep it updated.

Summary

A forking workflow is when developers fork an open-source repository, make changes in their fork, and then submit a Pull Request to merge their commits into the main branch.
