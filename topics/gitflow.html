Gitflow Workflow

A gitflow workflow is a large-scale branching model that orchestrates work into multiple branches for structured and cohesive development. It uses several branch types to separate features, releases, hotfixes, and the main development branch.

How it Works

  • 1. Main (main or master) branch — contains only production-ready code.
  • 2. Develop branch — integration branch where features are merged before release.
  • 3. Feature branches — branched from develop for new features.
  • 4. Release branches — branched from develop to prepare a new production release.
  • 5. Hotfix branches — branched from main to quickly fix production issues.
  • 6. Merge back into develop and/or main when complete.

When to Use

  • Medium to large projects with multiple developers.
  • Projects requiring clear release and hotfix management.
  • Teams using CI/CD pipelines and code review processes.

Common Workflow Command

# Start a new feature branch from develop
$ git checkout develop
$ git checkout -b feature/new-dashboard

# Finish feature branch and merge back to develop
$ git checkout develop
$ git merge feature/new-dashboard

# Start a release branch
$ git checkout -b release/1.2.0 develop

# Finish release branch, merge into main and develop
$ git checkout main
$ git merge release/1.2.0
$ git checkout develop
$ git merge release/1.2.0

Advantages

  • Keeps production code stable.
  • Supports parallel development.
  • Organized release and hotfix process.

Disadvantages

  • Highly complex and much more demanding than other workflows.
  • Requires strict adherence to branching rules.

Summary

Gitflow is a structured workflow that uses multiple branches — like develop, feature, release, and hotfix — which are merged into main only when ready for production.
