Proper Pull Requests

A Pull Request (PR) is a formal way to propose changes from one branch (or fork) into another. Proper conventions help maintain clarity, reviewability, and collaboration efficiency.

Best Practices

  • 1. Clear Title: Summarize the change in a few words.
  • 2. Descriptive Description: Explain what was changed and why. Include context for reviewers.
  • 3. Reference Issues: Link any relevant GitHub issues with #issue-number.
  • 4. Use Drafts if Needed: Indicate incomplete work with a draft PR.
  • 5. Review Before Merging: Ensure all tests pass and code is clean before approval.

Why it Matters

  • Improves readability and collaboration.
  • Reduces errors when merging into main branches.
  • Provides a clear history of changes for future reference.

Common Examples

Title: feature/add-login-validation

Added form validation for the login page.
- Checks for empty username or password fields.
- Displays error messages dynamically.
- Updated unit tests for validation.
Closes #42

Title: bugfix/fix-user-registration

Fixed issue where users could submit the registration form without a valid email.
- Added regex validation
- Updated error messages
- Tested on Chrome and Firefox
Closes #37

Summary

Maintaining clean and proper pull requests are crucial for sustainable work.
